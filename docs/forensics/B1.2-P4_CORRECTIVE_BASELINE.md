# B1.2-P4 Corrective Baseline (H01-H04)

Date: 2026-03-03  
Branch context audited: `b12-p3-pg-stampede-corrective` (post-PR154 merge state)

## Scope

- Token format + hashing boundary in refresh lifecycle service
- Tenant selection determinism in login issuance flow
- Reuse behavior for rotated/revoked refresh tokens
- Test inventory for attacker-first theft scenario
- CI wiring for merge-blocking P4 proof and negative controls

## H01 - Family revocation is not implemented on refresh-token reuse

- Verdict: **Validated**
- Evidence pointers:
  - `backend/app/services/auth_tokens.py`
    - `rotate_refresh_token(...)` acquires row lock and returns `None` when token is rotated/revoked/expired/mismatch.
    - No code path revokes family (`family_id`) when replayed rotated token is presented.
    - No DB mutation for successor tokens on reuse failure.
  - `backend/tests/test_b12_p4_token_lifecycle.py::test_refresh_rotation_single_use`
    - Asserts only 401 on reuse of A and row count stability; does not assert successor-family invalidation.
- Conclusion:
  - Current behavior is “single-use reject,” not “incident response revoke family.”

## H02 - Tenant selection at issuance is not mandatory/deterministic

- Verdict: **Validated**
- Evidence pointers:
  - Contract allows optional tenant_id:
    - `api-contracts/openapi/v1/auth.yaml` (`/api/auth/login` request body: required `email`, `password`; `tenant_id` optional)
    - `api-contracts/dist/openapi/v1/auth.bundled.yaml` mirrors this.
  - Generated model reflects optional tenant_id:
    - `backend/app/schemas/auth.py::LoginRequest.tenant_id: Optional[UUID] = None`
  - Runtime flow accepts omitted tenant_id and infers membership:
    - `backend/app/api/auth.py::login(...)` -> `resolve_tenant_membership(... requested_tenant_id=request.tenant_id)`
    - `backend/app/services/auth_tokens.py::resolve_tenant_membership(...)`
      - If `requested_tenant_id` is `None`, query with `LIMIT 2`; returns tenant when exactly one active membership, else `None`.
- Conclusion:
  - Tenant binding is not contract-mandatory at issuance.

## H03 - Hashing is applied to full token blob, not secret-only

- Verdict: **Validated**
- Evidence pointers:
  - `backend/app/services/auth_tokens.py`
    - `_refresh_token_digest(refresh_token: str)` computes SHA-256 over entire token string.
    - `hash_refresh_token_for_storage(refresh_token)` bcrypt-hashes digest of entire token.
    - `verify_refresh_token(refresh_token, token_hash)` compares digest of entire token.
  - Token format includes selector components:
    - `_mint_refresh_token_value(...)` => `"<tenant_id>.<token_id>.<secret>"`
    - `parse_refresh_token(...)` parses tenant_id and token_id.
- Conclusion:
  - Hash input is full blob (`tenant.token.secret`), not secret component only.

## H04 - Tests do not include attacker-first theft scenario

- Verdict: **Validated**
- Evidence pointers:
  - `backend/tests/test_b12_p4_token_lifecycle.py` contains:
    - login issuance checks
    - single-use rotation/reuse 401
    - tenant mismatch rejection
    - plaintext-at-rest check
    - concurrency one-success/one-fail check
    - signing hot-path no secret-fetch check
  - Missing scenario:
    - No test for: attacker uses A->B, legitimate later reuses A, then B must fail due family revocation.
- Conclusion:
  - Existing suite cannot prove theft-response semantics.

## Current CI wiring (merge-blocking jobs)

- Positive P4 proof path:
  - `.github/workflows/ci.yml` job `jwt-tenant-context-invariants`
  - Executes `pytest backend/tests/test_b12_p4_token_lifecycle.py -q` (merge-blocking in CI matrix).
- Negative-control path:
  - `.github/workflows/ci.yml` job `phase1-negative-controls`
  - Executes `scripts/contracts/run_negative_controls.sh` (merge-blocking).
  - Current P4 negatives present:
    - `SKELDIR_B12_P4_DISABLE_ROTATION_UPDATE` (single-use failure expected)
    - `SKELDIR_B12_P4_STORE_PLAINTEXT` (plaintext-at-rest failure expected)
  - Missing negatives for corrective scope:
    - disable family revocation on reuse
    - hash full blob instead of secret-only
    - tenant-less login accepted for multi-tenant user

## Baseline conclusion

P4 currently satisfies rotation mechanics and bounded lookup but fails three corrective invariants required by this directive:

1. No family revocation incident response on token reuse.
2. Tenant determinism not contract-mandatory at login issuance.
3. Refresh hash boundary implemented on full token blob rather than secret-only.

These are now the direct remediation targets.
