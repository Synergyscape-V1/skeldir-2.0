# B1.2-P4 Context Baseline (H01-H06)

Date: 2026-03-02  
Branch context: local `main` workspace

## Scope
- Contracts audit for `/api/auth/login` and `/api/auth/refresh`
- Implementation inventory for auth handlers and token helpers
- Schema inventory for refresh token persistence
- Secrets/key usage inventory for signing path boundedness
- CI wiring inventory for merge-blocking tests and negative controls

## H01 - Auth endpoints exist only as stubs / sample implementations
- Verdict: **Validated**
- Evidence:
  - `backend/app/api/auth.py` explicitly labels login/refresh/logout as "sample implementation for contract enforcement testing."
  - `backend/app/api/auth.py` login returns hardcoded `sample_access_token` / `sample_refresh_token`.
  - `backend/app/api/auth.py` refresh returns hardcoded `new_access_token` / `new_refresh_token`.

## H02 - No refresh token persistence store exists (or plaintext violation)
- Verdict: **Validated (store absent)**
- Evidence:
  - No auth refresh token ORM model found under `backend/app/models/`.
  - No migration creating auth refresh token table found under `alembic/versions/`.
  - `rg` hits for `refresh_token` storage map to platform credentials (`encrypted_refresh_token`) only:
    - `backend/app/models/platform_credential.py`
    - `alembic/versions/007_skeldir_foundation/202601281200_b060_phase2_platform_connections.py`
  - No auth lifecycle fields found (`token_hash`, `rotated_at`, `replaced_by_id`, `family_id`, `revoked_at`) outside platform credential context.

## H03 - Tenant-scoping for refresh tokens is undefined
- Verdict: **Validated**
- Evidence:
  - No auth refresh token table/model exists, therefore no enforceable `(tenant_id, user_id)` binding substrate.
  - Current `POST /api/auth/refresh` handler in `backend/app/api/auth.py` does not reference tenant/user state from refresh token storage.

## H04 - Rotation-on-use is not race-safe under concurrency
- Verdict: **Validated**
- Evidence:
  - `backend/app/api/auth.py` refresh handler is stateless stub logic; no transaction, lock, or token state transition.
  - No persistence model or SQL path exists for `SELECT ... FOR UPDATE` / single-use token transition semantics.

## H05 - Issuance/signing path risks per-request secret retrieval
- Verdict: **Partial / needs P4 confirmation for issuance path**
- Evidence:
  - Verification/read path hardening exists in `backend/app/core/secrets.py` with in-proc/Postgres cache primitives and refresh debouncing.
  - Signing primitive exists in `backend/app/security/auth.py` via `mint_internal_jwt()` -> `get_jwt_signing_material()` (cached ring path), but auth login endpoint currently does not use it (stub), so production issuance boundedness is not yet proven.
  - P4 implementation must prove no per-request secret fetch reintroduced in login/refresh issuance.

## H06 - CI lacks non-vacuous tests + negative controls for rotation/plaintext-at-rest
- Verdict: **Validated for P4 scope**
- Evidence:
  - Existing negative control harness `scripts/contracts/run_negative_controls.sh` covers P0-P3 controls (contract drift, JWT/GUC, users substrate, verifier cache), but does not contain P4 controls for:
    - refresh rotate-on-use bypass
    - plaintext refresh token storage regression
  - CI `jwt-tenant-context-invariants` job (`.github/workflows/ci.yml`) runs P1-P3 tests, but no P4 lifecycle test module is wired.
