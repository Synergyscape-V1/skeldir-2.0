# B1.2-P5 Cost/GC Baseline (Pre-Corrective)

## Scope
- Directive: B1.2-P5 corrective action for hot-path I/O elimination + denylist GC.
- Baseline branch: `main` merge base `0ea68df55` (PR #158 merge commit).

## Hypothesis Adjudication

### H01 - Per-request DB I/O exists on JWT hot path
- **Status:** Confirmed.
- **Evidence:**
  - `backend/app/security/auth.py`
  - `assert_access_token_active(...)` opened `AsyncSessionLocal` and called `evaluate_access_token_revocation(...)` for every authenticated request.
  - `backend/app/services/auth_revocation.py`
  - `evaluate_access_token_revocation(...)` queried `public.auth_access_token_denylist` and `public.auth_user_token_cutoffs`.
  - `backend/app/tasks/context.py`
  - Worker envelope path called `assert_access_token_active(...)`, inheriting the same per-call DB behavior.

### H02 - No bounded periodic GC mechanism exists for denylist
- **Status:** Confirmed.
- **Evidence:**
  - `backend/app/tasks/maintenance.py` had no denylist cleanup task.
  - `backend/app/tasks/beat_schedule.py` had no scheduled denylist cleanup entry.
  - Schema had `expires_at` + index (`idx_auth_access_token_denylist_expires_at`) but no operational delete loop.

### H03 - Naive cache can break correctness under concurrency
- **Status:** Valid risk.
- **Evidence:**
  - Existing implementation had no cache substrate/eventing to propagate revocation updates across runtimes.
  - Direct in-process memoization without synchronization/event propagation would miss revocations.

### H04 - Worker parity can regress if revocation moves out of DB
- **Status:** Valid risk.
- **Evidence:**
  - Worker revocation path was coupled to HTTP verifier function in `backend/app/tasks/context.py`.
  - Any cache-only move in HTTP without shared path would diverge worker behavior.

## Route-Family Boundary Check
- JWT auth is dependency-bound (`Depends(get_auth_context)`) on bearer routes.
- Webhook/HMAC routes remain on `tenantKeyAuth`/signature path (`backend/app/api/webhooks.py` family), separate from JWT verifier.

## Propagation SLA Target
- Corrective SLA target set to **<= 5 seconds** across runtimes for revocation event visibility.
- Rationale: near-immediate revocation while avoiding per-request DB I/O.

## Memory Bounds (Operational Estimate)
- Access tokens are 15-minute lifetime; denylist cache is TTL-bounded by `expires_at`.
- Cache footprint scales with active revoked JTIs within current token lifetime window + `(tenant_id,user_id)` cutoff map.
- Corrective design requires eviction of expired JTIs and bounded clean-token memoization by token `exp`.
