## Root-Cause Findings

1. The provider registry existed, but only as a revenue-fetch abstraction; lifecycle semantics were missing.
2. Capability truth was already centralized in P0, but adapter-layer declarations did not exist and could drift.
3. Without a deterministic adapter, P5 proofs would be pressured toward flaky live-provider tests.
4. P5 had no independent adjudication context in CI/required checks before this remediation.

## Exact Remediations Performed

1. Generalized existing `ProviderRegistry` to a provider-keyed generic abstraction (still consumed by realtime revenue path).
2. Added canonical OAuth lifecycle adapter module:
   - lifecycle request/response dataclasses,
   - `OAuthLifecycleAdapter` protocol,
   - capability model + declarations,
   - `OAuthLifecycleRegistry`,
   - deterministic reference adapter (`dummy`),
   - runtime scaffold adapter (`stripe`, explicitly deferred to P6/P7).
3. Added provider-neutral lifecycle dispatcher service (`ProviderOAuthLifecycleDispatcher`) using registry dispatch for all lifecycle operations.
4. Added P5 capability governance contract: `contracts-internal/governance/b13_p5_oauth_adapter_capabilities.main.json`.
5. Added enforcement gate: `scripts/ci/enforce_b13_p5_adapter_layer.py` for:
   - required context presence,
   - canonical method completeness,
   - deterministic adapter presence/registration,
   - registry-backed dispatch (no provider branching),
   - capability alignment with P0 matrix and P5 contract,
   - phase-collapse guard (no runtime platform-oauth router in P5).
6. Added P5 non-vacuous proof tests: `backend/tests/test_b13_p5_adapter_layer.py`.
7. Added CI job `B1.3 P5 Adapter Layer Proofs` and added it to required-check governance contract.
