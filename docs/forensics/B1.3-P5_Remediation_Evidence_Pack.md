## Root-Cause Findings

1. Registry abstraction existed but OAuth lifecycle semantics did not.
2. Capability truth needed explicit adapter-layer contract and CI coupling to prevent scope drift.
3. Deterministic in-repo lifecycle adapter was required for non-flaky proofs.
4. Review governance + collaborator topology created an operational deadlock for merge approval (single author == single code owner).

## Exact Remediations Performed

1. Generalized `ProviderRegistry` for provider-keyed generic reuse while preserving realtime revenue behavior.
2. Added canonical OAuth lifecycle adapter module with method-complete protocol, request/response models, capability declarations, registry, deterministic adapter (`dummy`), and scaffold runtime adapter (`stripe`).
3. Added provider-neutral dispatcher (`ProviderOAuthLifecycleDispatcher`) that resolves adapters via registry for all lifecycle methods.
4. Added P5 capability governance contract: `contracts-internal/governance/b13_p5_oauth_adapter_capabilities.main.json`.
5. Added structural enforcement script: `scripts/ci/enforce_b13_p5_adapter_layer.py`.
6. Added non-vacuous test suite with negative controls: `backend/tests/test_b13_p5_adapter_layer.py`.
7. Added CI context `B1.3 P5 Adapter Layer Proofs` in `.github/workflows/ci.yml`.
8. Added required-check governance inclusion for P5 context in `contracts-internal/governance/b03_phase2_required_status_checks.main.json`.
9. Updated branch protection required checks on `main` so `B1.3 P5 Adapter Layer Proofs` is required.
