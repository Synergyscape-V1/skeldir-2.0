## 4. Non-Vacuity Statements for Each Negative Control

### NC-1: Cross-tenant RLS read isolation (EG9.2)
**Control:** A-token reads A revenue row; B-token reads B revenue row; both totals are distinct (111.00 vs 222.00 cents/100). Test asserts `float(read_a["total_revenue"]) != pytest.approx(seeded.revenue_b_total)` and vice versa.
**How it would fail under regression:** If RLS policy were dropped or broken, `read_a["total_revenue"]` would return the wrong (or null) value; the `pytest.approx` assertion on the correct value would fail, and/or the cross-inequality check would fail. The seeding uses distinct, non-equal values (111.00 vs 222.00) precisely to make any leakage detectable via value assertion, not just status-code assertion.

### NC-2: Cross-tenant write denial (EG9.2)
**Control:** A-token attempts to elevate tenant-B-only user to admin via `POST /api/auth/admin/membership-role`; asserts `status_code == 404`.
**How it would fail under regression:** If RLS or route guard were removed, the write might succeed (2xx) or return a wrong status; the `assert == 404` would fail. The 404 posture is intentional hide-existence behavior (not a missing route), as confirmed by the fact that the route exists and succeeds for in-tenant targets (the admin_ok_response test proves the route is reachable with admin credentials).

### NC-3: Worker tenantless task failure before side effects (EG9.3)
**Control:** Task dispatched with no `AUTHORITY_ENVELOPE_HEADER`; `missing_envelope_result.get(propagate=True)` raises `Exception(match="authority_envelope header is required")`; `_count_worker_effect_rows(tenant_id=…, task_id=missing_envelope_result.id) == 0`.
**How it would fail under regression:** If the authority envelope check were removed from the task, the task would execute without a tenant context, no exception would be raised (breaking `pytest.raises`), and/or the side-effect row would be written (breaking `== 0`). Both assertions must pass independently.

### NC-4: Invalid webhook signature rejection (EG9.4)
**Control:** Webhook sent with `digest = "0" * len(digest)` (all-zero HMAC); asserts canonical 401 via `_assert_non_leaky_problem`.
**How it would fail under regression:** If signature verification were bypassed, the invalid-digest request would return 200 (same as the valid-signature case), and `_assert_non_leaky_problem(…, expected_status=401)` would fail on `assert response.status_code == 401`. The tampered digest is computed in the test itself (not injected via mock), so no stub can mask a regression.

### NC-5: Forbidden substring bans in error responses (EG9.5)
**Control:** For each of the three failure modes, `_assert_non_leaky_problem` lowercases the full JSON body and asserts that none of the 12 forbidden substrings appear (e.g., `"stripe"`, `"signature"`, `"pyjwterror"`, `"invalid_signature"`).
**How it would fail under regression:** If the error canonicalizer regressed and leaked an internal exception message (e.g., a raw `pyjwt.exceptions.InvalidSignatureError` string), the substring `"invalidsignature"` or `"pyjwterror"` would appear in the lowercased body, causing the assertion to fail. The ban list is intentionally broad and includes both camelCase and snake_case variants to catch multiple serialization formats.
